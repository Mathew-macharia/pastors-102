## Original Readme Below (For reference only)

Replaces the direct treasury → 40 fan-out from `funder/` with a 5-stage
pipeline that interposes 16 fresh intermediate wallets and two Jupiter
swaps (SOL → USDC → SOL) between the treasury and the final 40 wallets.

This raises the bar against **automated clustering tools** (the kind most
trading bots and free explorers use). It does **not** defeat serious
forensics services (Chainalysis, TRM Labs, Arkham, Binance's own AML
platform). Read the [OPSEC reality](#opsec-reality-must-read) section
before using this in front of a CEX deposit.

**Total**: 72 transactions, 16 fresh intermediate wallets.
**Runtime**: ~90 minutes with default 30-300s random delays per tx.

prevents a CSV-order leak that could group wallets together visually.

## OPSEC reality (must read)

In **March 2026**, Arkham Intelligence and ChainCatcher publicly flagged a
**24,500 SOL transfer to Binance via relay addresses** — exactly the
multi-hop pattern this script implements. The deposit was traced and
identified within hours despite the relay structure. Reference: Arkham,
ChainCatcher coverage from March 2026.

What this script DOES buy you:

- Defeats free / open-source explorers' automatic "send/receive" clustering.
- Breaks the direct on-chain link between the treasury and the 40 final
  wallets — no single graph traversal connects them.
- Creates plausible-looking "DEX user activity" patterns (each intermediate
  wallet looks like a small Jupiter user).
- Adds 4 extra hops + 2 mint changes, increasing analyst time-to-cluster
  by ~10-50× depending on the tooling.

What it DOES NOT buy you:

- **CEX-grade anonymity.** Binance, Coinbase, Kraken all subscribe to TRM
  Labs and Chainalysis, which automatically follow Jupiter routes through
  USDC bridges. Multi-hop with swaps is a **flagged pattern** in those
  systems, not an obfuscation that works.
- **Time anonymity.** All 16 intermediates are created within ~90 minutes
  in a sequential pattern. A timing-clustering algorithm picks this up.
- **Ground-truth deniability.** If the destination is later linked to bad
  activity, the entire 5-hop chain leads back to your treasury.

If you're building this to deposit memecoin profits to Binance:
**this script does not protect you.** Use it for the funding leg only,
treat the on-ramp/off-ramp as a separate and harder problem.

## Prerequisites

1. Phase 1 done: `../wallets/wallets.csv` exists with the 40 final pubkeys.
2. Phase 2 (funder) treasury created: `../funder/treasury/treasury.json`.
3. Treasury funded with **at least ~6 SOL**:
   - 40 finals × 0.10 SOL avg = 4 SOL
   - + ~0.5 SOL slippage (1% × 2 swaps × 8 chains × 0.6 SOL each)
   - + ~0.05 SOL ATA rents (8 USDC ATAs × ~0.00204 SOL × 2 stages = ~0.033 SOL)
   - + ~0.005 SOL fees & tips
   - + 1 SOL safety buffer
   - **Withdraw 6 SOL from Binance to be safe.**
4. Helius API key (free tier works) in `.env`.

## Install — Ubuntu / WSL Ubuntu

```bash
cd multihop
bash setup.sh
```

`setup.sh` checks all prereqs, creates a `.venv/`, installs deps, creates
`intermediates/` (chmod 700), and copies `.env.example` to `.env`.

Edit `.env`:

```
HELIUS_API_KEY=your_key_here
```

(You can reuse the same key from `../funder/.env`.)

## Run

```bash
source .venv/bin/activate

# 1. Pre-flight: validates treasury balance, generates A/B wallets, allocates
#    final-wallet groups, but sends NO transactions.
python multihop.py --dry-run

# 2. Run a single stage to verify (recommended for first run):
python multihop.py --only-stage 1

# 3. Or run everything end-to-end:
python multihop.py

# 4. If something fails mid-run, just re-run -- already-confirmed
#    transactions are skipped automatically.
python multihop.py
```

## State machine

`state/multihop_state.json` tracks every chain at each stage. Each tx slot
has: `txid`, `slot`, `status` (`pending`/`sent`/`confirmed`/`failed`),
`error`, `completed_at_utc`.

Resumption rules:

- **Already `confirmed`**: skipped.
- **`failed` or `pending`**: retried on next run.
- **`sent` but not yet `confirmed`** (interrupted run): the next run
  builds a NEW transaction for that step (see caveat below).

> ⚠ **Caveat for stage 2/4 (Jupiter swaps)**: if a swap tx was submitted but
> not confirmed, manually verify on Solscan before re-running, because a
> Jupiter swap that lands late could double-swap your funds. Stages 1, 3, 5
> use `skipPreflight=true` and have idempotent retry semantics, so they are
> safer to re-run.

## Cost breakdown (default settings, 8 chains, SWQoS-only)

| Item                                              | Cost                          |
| ------------------------------------------------- | ----------------------------- |
| Stage 1: 8 × (transfer + tip + fee + priority)    | ~0.0001 SOL                   |
| Stage 2: 8 × Jupiter swap fee + 1% slippage       | ~0.04 SOL (slippage dominates) |
| Stage 3: 8 × (USDC transfer + ATA create rent)    | ~0.018 SOL (rent dominates)   |
| Stage 4: 8 × Jupiter swap fee + 1% slippage       | ~0.04 SOL (slippage dominates) |
| Stage 5: 40 × (transfer + tip + fee + priority)   | ~0.0005 SOL                   |
| **Overhead total**                                | **~0.1 SOL** (~$15 at ~$150/SOL) |
| Recipient amount to 40 finals                     | ~3.6 - 4.4 SOL (your money)   |

Slippage is the largest non-recipient cost. Tighten with `--slippage-bps 50`
(0.5%) if you want — but failed swaps cost you a re-run.

## Troubleshooting

| Symptom                                                          | Cause / Fix                                                                                            |
| ---------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| `treasury keypair not found at ../funder/treasury/treasury.json` | Run `../funder/setup.sh` first.                                                                        |
| `wallets.csv not found`                                          | Run `../wallets/generate_wallets.py` first.                                                            |
| `jupiter quote 429`                                              | Free-tier rate limit. Wait 60s and retry, or get a Jupiter API key.                                    |
| `B has 0 USDC atoms`                                             | Stage 3 didn't actually land. Check Solscan for the stage-3 txid in `state/multihop_state.json`.       |
| `confirmation_timeout` repeatedly                                | Network congestion. Bump `--prio-max 500000`. Re-run.                                                  |
| `state file treasury mismatch`                                   | You re-generated the treasury. Either restore the original treasury or delete `state/multihop_state.json` to start fresh. |
